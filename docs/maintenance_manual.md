# 维护手册

## 新增一套卷子
1. 把原始资料放进服务器或本地开发环境对应的 `raw/` 目录。
2. 运行 `scripts/register_raw_assets.py` 更新总表。
3. 按清洗包格式整理题目、答案和图片。
4. 运行校验和 dry-run。
5. commit 导入。
6. 如有成绩表，运行统计导入与难度计算。
7. 运行质量检查并补齐缺失项。
8. 如需共享，再导出半公开版本。

## 字段变更规则
- 任何新增字段必须同步更新 SQLite schema、字段字典、样例清洗包和测试。
- 字段名优先英文、语义稳定、避免临时简称。

## 数据修复规则
- 优先修复清洗包或脚本，不直接手改 SQLite。
- 如果必须手动修复，修复后需要补回对应清洗包并重新导入，保证可追溯。

## 服务器部署与同步
- 源码仓通过 `git pull` 同步到服务器，不直接同步生产数据库文件。
- 服务器使用环境变量或私有 `.env` 指向 `<SERVER_DB_PATH>`、`<SERVER_ASSETS_DIR>`、`<SERVER_RAW_DIR>`。
- 更新代码后按“拉代码 -> 更新虚拟环境依赖 -> 重启 API 服务”流程执行。
- 服务管理建议使用 systemd、supervisor 或等价守护进程，模板只保留占位符。

## 备份与回滚
- 生产 SQLite 与生产 assets 必须做定期备份。
- 每次大规模录入后导出一份内部 JSONL 快照并纳入备份。
- 回滚时优先回滚服务器上的数据库备份，再回滚源码仓版本。
- 生产事故排查时，不要把真实服务器路径或凭据写回 Git 历史。
